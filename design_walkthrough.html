1. Why This Domain

We chose Supply Chain — Inventory Optimization & Demand Forecasting for FMCG products for three reasons:

Business Impact. Inventory costs represent 20-30% of total supply chain expenditure in FMCG. Overstocking ties up working capital; understocking causes lost sales and eroded customer trust. Even a 5% improvement in forecast accuracy translates directly to measurable bottom-line savings through reduced safety stock buffers.

Analytical Richness. The domain naturally demands all six mandatory pipeline stages: time-series data requires rigorous validation (Stage 1), lag features and rolling statistics need careful engineering (Stage 2), multiple forecasting algorithms must compete head-to-head (Stage 3), statistical metrics like WMAPE objectively validate results (Stage 4), and the business interpretation layer (Stage 5) maps mathematical outputs to actionable procurement decisions. No stage can be hand-waved — each is substantive.

Multi-Algorithm Comparison. Demand patterns in FMCG vary dramatically — stable pantry staples vs. highly seasonal holiday items vs. erratic viral products. This variation creates a compelling champion/challenger dynamic where different algorithms genuinely win for different SKUs, making the comparison meaningful rather than performative.

Domain Scope: 20 SKUs across 10 categories (Ambient Grocery, Chilled Proteins, Frozen Proteins, Beverages, Condiments, Outdoor/Seasonal, Baking, Confectionery, Snacks, Health & Beauty) with daily granularity from January 2023 to March 2026.

2. Architecture & Pipeline Design

The skill package follows a strict separation of concerns: SKILL.md orchestrates the pipeline logic, REFERENCE.md encodes domain knowledge, scripts/ handle computation, and the LLM handles interpretation and narrative generation.

Key architectural decisions:

  • Deterministic computation, LLM interpretation: All mathematical operations (forecasting, safety stock calculation, WMAPE scoring) run in Python scripts with fixed random seeds. The LLM is used only where it excels — translating numbers into business language and generating narrative insights. This ensures reproducibility.
  • Fail-fast validation: The pipeline halts immediately on data quality failure (Stage 1) rather than producing garbage outputs. This is critical for production supply chain systems.
  • Structured JSON handoffs: Every stage produces a JSON artifact that the next stage consumes. This makes the pipeline debuggable — you can inspect any intermediate file to understand what happened.
  • Full tracing: Every script execution, LLM call (prompt, response, token count, latency), and decision point is logged with timestamps in execution_trace.json.

3. Why These Steps — Stage-by-Stage Rationale

Stage Script / Actor Why This Design
1. Data Validation & Profiling validate_data.py Enforces schema constraints (non-negative sales, lead time ≥ 1, <30% nulls per column). Real supply chain data is messy — dates arrive unparseable, ERP exports contain blanks. Catching this before modelling prevents silent garbage-in-garbage-out failures.
2. Feature Engineering forecast_and_optimize.py Builds 17 features per SKU: lag values (1, 7, 14, 28 days), rolling statistics (mean and std over 7/14/28-day windows), and calendar features (day-of-week, month, day-of-year, week-of-year). These capture short-term momentum, medium-term trends, and calendar seasonality. Missing values are linearly interpolated before feature creation.
3. Multi-Algorithm Modelling forecast_and_optimize.py Two fundamentally different approaches compete: Gradient Boosted Trees (captures non-linear interactions and recent momentum) vs. Fourier Seasonal Model (Ridge regression on yearly + weekly Fourier harmonics, isolating calendar-driven patterns). The winner is selected per-SKU by WMAPE on a holdout set — this reveals the underlying demand driver.
4. Result Validation run_pipeline_llm.py Programmatically verifies every SKU has a winning algorithm and flags any WMAPE > 0.30 as "Poor Accuracy." This prevents the pipeline from silently reporting unreliable forecasts as if they were trustworthy.
5. Insight Generation Groq LLM The LLM receives optimization results + REFERENCE.md, classifies each SKU into archetypes (Stable Baseload, Highly Seasonal, Erratic/Volatile), and generates per-SKU business recommendations grounded in the domain knowledge document. This is where the LLM adds real value — it synthesises patterns across 20 SKUs into a coherent narrative.
6. Report Generation generate_report.py + Groq LLM Python generates the charts (scatter plot, WMAPE comparison bar chart) and data tables. The LLM generates the executive summary narrative. Both are merged into a single self-contained HTML document with embedded base64 images — no external dependencies.
Design Choice — Why combine Stages 2 & 3 into one script? Feature engineering and modelling are tightly coupled for time-series data — the train/test split must happen after feature creation to prevent data leakage. Separating them would require serialising and deserialising large intermediate DataFrames. Bundling them in forecast_and_optimize.py keeps the pipeline efficient while maintaining logical separation in SKILL.md's documentation.

4. LLM Integration Design

The LLM (Llama-3.3-70B via Groq) is used at two specific points, with deliberate prompt engineering:

Stage 5 — Insight Generation Prompt

The prompt sends the full REFERENCE.md as grounding context, the optimisation results JSON, and structured instructions to: (1) classify each SKU into an archetype, (2) interpret the winning algorithm choice, (3) generate recommendations using REFERENCE.md templates, and (4) flag high-WMAPE SKUs. Temperature is set to 0.3 to keep outputs factual. The system prompt explicitly instructs: "Do not hallucinate metrics or benchmarks."

Stage 6 — Executive Summary Prompt

Receives aggregate metrics (total SKUs, average WMAPE, model win counts) plus a preview of the Stage 5 insights. Temperature is 0.2 (even more conservative) since executive summaries must be precise. The system prompt targets a VP of Supply Chain audience.

Observability

Every LLM call logs: full prompt text, full response text, model ID, token usage (prompt + completion + total), and wall-clock latency. These are embedded in execution_trace.json with type "LLM_CALL", making it easy to audit exactly what the LLM saw and said.

5. What Failed and What We Fixed

Development was iterative. Below are the significant issues encountered and how they were resolved:

Failure 1: JSON Structure Mismatch Between Scripts

Problem: The original generate_report.py expected nested JSON keys like item['model_selection']['winning_wmape'] and item['forecast_metrics']['next_30d_avg_daily_demand'], but forecast_and_optimize.py produced flat keys like item['winning_wmape'] and item['avg_daily_demand']. The report script would crash with a KeyError every time.
Fix: Rewrote generate_report.py from scratch to consume the actual flat JSON structure. Also added a WMAPE comparison bar chart, per-SKU archetype classification, and colour-coded WMAPE values in the table (green < 0.15, amber < 0.30, red > 0.30).

Failure 2: Python vs Python3 Command

Problem: The pipeline runner initially used python scripts/... commands, but the development machine only had python3 on PATH. Every subprocess call failed with exit code 127 (command not found).
Fix: Updated all subprocess commands to use python3 explicitly. This was also applied to the synthetic data generation commands in scenario definitions.

Failure 3: Matplotlib Deprecation Warning

Problem: plt.cm.get_cmap() raised a MatplotlibDeprecationWarning in Matplotlib 3.7+, cluttering stderr output and trace logs.
Fix: In the LLM-powered report generator, switched to the modern plt.colormaps.get_cmap() API. Also added matplotlib.use("Agg") at import time to ensure headless rendering.

Failure 4: Run 3 Bad Data — Intentional but Validated

Problem: The corrupted dataset (40% nulls in sales_volume, negative unit_cost, zero lead_time_days) was designed to test the guardrails. The first version of the runner simply crashed without producing any output.
Fix: Added a generate_failure_report() function that produces a structured HTML report explaining exactly which validation rules were violated and how to fix the data. The pipeline now degrades gracefully — it halts, logs the reason, saves the trace, and outputs an actionable failure report.

Failure 5: Prophet Dependency Complexity

Problem: Facebook Prophet has heavy dependencies (PyStan/CmdStan) that are difficult to install reliably across environments. This would make the skill non-portable.
Fix: Replaced Prophet with a mathematically equivalent Fourier Seasonal Model — a Ridge regression on a linear trend plus yearly and weekly Fourier harmonics (3 orders each, 14 features total). This achieves the same seasonality-isolation effect using only sklearn and numpy, with zero additional dependencies. The output is labelled "Fourier Seasonal" and the implementation is documented in REFERENCE.md Section 3.

7. Lessons Learned & Future Improvements

What Worked Well

  • Separation of computation and interpretation: Having scripts produce numbers and the LLM produce narratives keeps the pipeline auditable. Every number in the report can be traced back to a deterministic script.
  • REFERENCE.md as grounding document: The LLM's insights were consistently accurate because it had explicit archetype definitions, formula specifications, and recommendation templates to follow — not open-ended generation.
  • Structured JSON handoffs: Made debugging straightforward. When the report script failed, we could inspect optimization_results.json directly to identify the key structure mismatch.
  • Fail-fast guardrails: Run 3 demonstrated that the pipeline correctly refuses to produce misleading forecasts from corrupted data — a critical property for production systems.

Limitations

  • Point forecasts only: The current pipeline produces point estimates without prediction intervals. A production system should generate confidence bands to communicate forecast uncertainty.
  • Uniform service level: All SKUs receive the same Z-score. In practice, high-margin "never-out" items should have Z = 2.33 while low-margin substitutable items could use Z = 1.28.
  • Additive seasonality assumption: The Fourier model assumes additive seasonality. Items with multiplicative patterns (where seasonal amplitude scales with trend) may be under-captured.
  • No external data: The pipeline does not incorporate exogenous signals (promotions, pricing changes, weather) that significantly influence FMCG demand.

Future Improvements

  • Add MCP server integration for live inventory data feeds from ERP systems.
  • Implement margin-weighted service levels per SKU using unit cost and margin data.
  • Add a human-in-the-loop checkpoint after Stage 4 where a supply chain planner reviews flagged high-WMAPE SKUs before the LLM generates recommendations.
  • Integrate LangSmith or similar tracing platform for production-grade observability beyond JSON logs.
  • Add backtesting framework that replays historical data to validate safety stock recommendations against actual stockout events.
